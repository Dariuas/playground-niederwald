Security Fixes Applied This Session

All 7 fixes are verified in the passing build. Zero regressions introduced.

1
Auth guard on GET /api/admin/pavilions
Pavilion pricing and configuration data was readable by anyone without authentication.
app/api/admin/pavilions/route.ts
2
Auth guard on GET /api/admin/products
Full product catalog (names, prices, Square IDs) was publicly exposed without a session cookie.
app/api/admin/products/route.ts
3
Auth guard on POST /api/admin/products
Unauthenticated POST could create arbitrary products and trigger Square catalog API calls.
app/api/admin/products/route.ts
4
Auth guard on POST /api/admin/products/[id]/sync
Square sync endpoint was unauthenticated — anyone could trigger paid Square API operations.
app/api/admin/products/[id]/sync/route.ts
5
Auth guard on GET /api/admin/schedule
Day-of-week pricing schedule was readable without authentication.
app/api/admin/schedule/route.ts
6
HTML escaping (XSS) in email templates
All user-supplied fields (customer name, email, phone, message, subject, order numbers) were injected directly into HTML email bodies without escaping. A customer named <img src=x onerror=...> would execute in the recipient's mail client. An esc() function now sanitizes every user value before HTML insertion.
lib/emailTemplates.ts — affects all 5 email templates
7
Server-side input validation on reservation route
The API accepted any duration value (someone could book 365 hours and pay $0 due to calculation overflow), no email format validation, and no date bounds. Now enforces: duration 1–12 hours, valid email regex, date must be today through 6 months out.
app/api/reservation/route.ts

Open Security Issues

IssueSeverityRecommended Fix
Booking race condition
app/api/reservation/route.ts		 🔴 Critical Add UNIQUE(pavilion_id, date, start_time, end_time) constraint to pavilion_bookings in Supabase. The route will receive a unique violation on the second insert and return a natural 409.
No CSRF protection on admin mutations 🔴 Critical Generate a CSRF token in admin layout, include as hidden field in all forms, validate in POST/PATCH/DELETE handlers. Set session cookie to SameSite=Strict.
No rate limiting on public APIs 🟡 Medium Add next-rate-limit or Netlify edge rate limiting. Suggest: 3 reservation attempts/hr/IP, 5 contact form submissions/hr/IP.
Admin session maxAge = 30 days 🟡 Medium Reduce to 7 days in app/api/admin/login/route.ts. Reduces stolen-cookie window.
Admin password "admin123" 🟡 Medium Change to a strong password in .env.local and Netlify environment settings before launch.
